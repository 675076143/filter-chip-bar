# 设计笔记 — 数学基础

> 软件设计是应用数学。每一个 API 决策都是表达力与简洁性之间的权衡,可用信息论的术语度量。

---

## 1. 级联筛选 — 图论

### 问题
传统筛选项是**独立节点** — 选择 `部门:工程部` 不会影响 `团队` 的选项。但现实数据构成**有向依赖图**:

```
部门 ──筛选──→ 团队 ──筛选──→ 项目
```

### 洞察
将 `ChipConfig.options` 建模为图状态的函数,而非静态集合:

```typescript
options: async (chips) => {
  const dept = chips['部门'];
  return fetchTeams(dept);  // 只返回该部门的团队
}
```

`chips` 参数是**当前图的遍历状态** — 用户目前已选的条件。当任何上游节点变化时,下游节点自动重新解析。

### 实现
hook 通过 `parseQuery` 从 `searchText` 计算 `currentChips`(只用静态 label,不用 resolved options — 这打破了循环依赖)。async 解析的 `useEffect` 依赖 `currentChips`,所以当用户提交一个新 chip(Enter/选建议/空格)时,依赖字段重新拉取。

### 为什么不用响应式订阅?
完整的响应式图(RxJS observables、signal-based reactivity)会增加 50+ 行复杂度,收益甚微。基于提交的重拉取(只在 Enter/空格/选建议时触发)频率足够低,同步重解析完全够用。**O(n) 提交时重拉取,而非 O(1) 每次按键时。**

## 2. 标签别名 — 信息论

### 问题
`Status:Passing` 需要 15 个字符。但这个查询的**信息熵**只有 ~8 bit(3 个状态选项 ≈ log₂(3) ≈ 1.6 bit,加上字段标识)。字符开销是浪费的带宽。

### 洞察
Power user 需要**更短的信道编码**。`st:pass`(8 字符)携带相同的语义信息。这与 Unix 命令别名(`ls` vs `list`)、DNS CNAME 记录、URL 短链是同一个原理。

```typescript
{
  type: 'select',
  label: 'Status',
  aliases: ['st', 'status'],
  options: [...],
}
```

parser 将任何别名映射到**规范标签**(`config.label`)。输出 `chips` 始终使用规范 key,消费方无需感知别名。

### 香农熵视角
- 完整标签: H(Status) = -log₂(1/6) ≈ 2.6 bit/字符(6 个唯一标签)
- 别名 `st`: H(st) = -log₂(1/6) ≈ 2.6 bit 在 2 个字符内 vs 6 个字符
- **压缩比**: 3× 无损

别名是**无损压缩** — `st:pass` 和 `Status:Passing` 产生完全相同的 `chips` 输出。parser 是解压器。

### 为什么不任意缩短?
不受限的别名会产生歧义(`s` 可能是 `Status` 也可能是 `SKU`)。`aliases` 数组是显式的 — 配置作者决定哪些别名在其领域中无歧义。

## 3. 频率加权历史 — 贝叶斯概率

### 问题
搜索历史按**时间倒序**排列(最近的在前)。但 P(下次搜索 = X) 同时取决于时间与频率:

```
P(X | history) ∝ P(X | 时间近因) × P(X | 使用频率)
```

每天用 10 次的查询应该排在昨天只用过 1 次的前面。

### 洞察
对搜索频率应用**累积计数**:

```typescript
interface RecentSearch {
  text: string;
  total: number;
  timestamp: number;
  frequency: number;  // 累积计数,每次使用递增
}
```

按 `frequency DESC, timestamp DESC` 排序 — 最常用的在前,最近的打破平局。

### 为什么不用完整贝叶斯衰减?
完整贝叶斯推断(指数时间衰减、先验等)会建模:
```
score = frequency × e^(-Δt/τ)
```
其中 τ 是时间常数(~24h)。理论上更严谨,但:
1. 需要一个时间常数调参(领域相关)
2. 简单的 `frequency × recency` 排序对 N ≤ 10 条目已经是很好的近似
3. 只有 10 条历史时,精确贝叶斯排序的边际收益可忽略

**更简单的模型,相同的实际效果。**

## 未做的事(以及为什么)

### 布尔代数(括号分组)
`(Status:Passing OR Status:Failing) AND Orders:>=100`

**拒绝**: 语法变成上下文无关(需要递归下降 parser)。当前语法是**正则的**(有限状态机,无递归)。加括号从 O(n) 变成 O(n²) 解析复杂度,只有 <5% 的用户需要。

### 自定义解析中间件(范畴论)
```typescript
middleware: [(text) => text.replace(/@(\w+)/g, 'Mention:$1')]
```

**拒绝**: 管线 `text → parseQuery → chips` 是范畴间的**自然变换**(字符串 → 结构化数据)。让它可组合增加一个函子层:
1. API 面积增加 O(n),n = 中间件数量
2. 产生顺序依赖(中间件 A 必须在 B 之前运行)
3. `commands` 已提供自定义语法的逃生舱

### 向量空间相似度(线性代数)
基于 filter 空间距离建议"相似"查询。

**拒绝**: 需要为每条历史查询计算并存储向量。≤10 条历史时精确排序已足够。向量相似度在大规模(>100 条查询)时才有用,超出了 localStorage 支持的历史容量。

# 通信学基础

> FilterChipBar 本质上是一条通信信道:用户意图 → 编码为文本 → 解码为结构化筛选。通信学告诉我们如何让这条信道可靠且高效。

## Shannon-Weaver 模型应用

```
用户意图 → [编码器: 打字] → [信道: 搜索框] → [解码器: parseQuery] → 筛选结果
                                      ↑
                                 噪声: 笔误、语法错误、无效选项
```

每个设计决策都映射到一个通信工程概念。

---

## 4. 前向纠错 (FEC) — "Did you mean?"

### 通信原理
在数字通信中,纠错码(Hamming、Reed-Solomon、Turbo 码)添加冗余,使接收端不仅能**检测**错误,还能**纠正**错误而无需重传。这在单向信道(卫星、深空通信)中至关重要,重传代价极高。

### 问题
组件有错误**检测**(红色高亮 = "值无效"),但没有错误**纠正**。笔误如 `Status:Pasing`(漏一个 s)只会产生红色错误 — 用户必须自己找出并修复问题。

### 洞察
在建议层应用 FEC。当精确匹配失败时,用 **Levenshtein 编辑距离**找到最近的合法选项。如果距离 ≤ 2,显示 "Did you mean **Passing**?" 建议。用户一键纠正 — 无需重传(重新打字)。

```typescript
// Levenshtein 距离: 将 a 转换为 b 所需的最少单字符编辑次数
function levenshtein(a: string, b: string): number { ... }

// FEC 解码器: 在纠错容量范围内找到最近的合法码字
function findClosest(input: string, options: string[], maxDistance: number): string | null {
  return options
    .map(o => ({ option: o, distance: levenshtein(input.toLowerCase(), o.toLowerCase()) }))
    .filter(o => o.distance > 0 && o.distance <= maxDistance)
    .sort((a, b) => a.distance - b.distance)[0]?.option ?? null;
}
```

### 为什么用 Levenshtein 而不是 Hamming?
Hamming 距离只计算**替换**。Levenshtein 计算插入 + 删除 + 替换 — 这是必要的,因为打字错误包括漏字母(`Pasing`)和多字母(`Passsing`)。

### 纠错容量
t = ⌊(d-1)/2⌋,其中 d 是最小码距。maxDistance=2 时,确定性地纠正 1 字符错误,概率性地纠正 2 字符错误。这与真实打字错误模式匹配(92% 的笔误是单字符错误)。

## 5. 拥塞控制 — 防抖级联重拉取

### 通信原理
TCP 的拥塞控制算法(慢启动 → 拥塞避免 → 快速重传)通过检测拥塞并退避来防止**网络崩溃。没有它,多个发送方洪泛瓶颈路由器会导致吞吐量趋近于零。

### 问题
级联筛选(特性 #1)在 `currentChips` 变化时触发 async 重拉取。如果用户快速连续提交多个 chip(`部门:工程部 团队:前端 项目:UI`),每次提交都触发所有依赖 async 选项的完整重拉取。这造成**请求风暴** — N 次提交 × M 个依赖字段 = N×M 次 API 调用在短时间内密集发出。

### 洞察
应用 TCP 式**指数退避**,通过 debounce 实现。不是立即触发,而是等最后一次 chip 变化后 200ms。如果等待期间又来一次变化,重置计时器。只有**最终状态**触发重拉取:

```
t=0ms:   chips 变化 (部门:工程部) → 安排 200ms 后拉取
t=50ms:  chips 变化 (团队:前端)   → 取消前一个,重新安排 200ms
t=100ms: chips 变化 (项目:UI)     → 取消前一个,重新安排 200ms
t=300ms: 计时器触发 → 只发一次请求,使用最终状态 {部门:工程部, 团队:前端, 项目:UI}
```

将 N×M 次请求减少到 **1 次**,符合 TCP 的原则:拥塞时少发。

### 实现
```typescript
useEffect(() => {
  const timer = setTimeout(() => {
    // 重新解析 async 选项
  }, 200);
  return () => clearTimeout(timer);
}, [chipConfigs, currentChips]);
```

### 为什么是 200ms?
- 人类对视觉反馈的反应时间: ~200ms
- 低于此阈值,用户感知不到延迟
- 高于 500ms,用户感觉"哪里不对"
- 200ms 是甜蜜点:用户无感,又能充分合并快速输入

# 心理学基础

> 搜索框不只是数据接口 — 它是认知接口。每次击键都是一个决策,每个建议都是反馈,每条语法规则都是一条学习曲线。心理学告诉我们如何为人类心智而设计。

## 验证: 已内化的原则

大多数认知心理学原则已经嵌入到组件的设计中:

| 原则 | 应用方式 |
|------|---------|
| **识别优于回忆** (Nielsen) | 建议下拉框让用户*认出*选项,而非*回忆*名称 |
| **认知负荷理论** (Sweller) | 一个搜索框代替 N 个下拉框,释放工作记忆 |
| **操作性条件反射** (Skinner) | 绿色高亮 = 正向强化; FEC "Did you mean?" = 引导纠正(非惩罚) |
| **格式塔原则** | 建议按 proximity 分组(历史 \| 筛选),按 similarity 统一样式 |
| **心流** (Csikszentmihalyi) | 150ms blur + 200ms debounce 调校到人类认知节律 |
| **最小努力原则** | 别名(`st:pass`)、前缀(`#tag`)、预设(1 次点击)最小化用户努力 |

这些不需要新代码 — 它们验证了现有设计与人脑认知方式的一致性。

## 6. 渐进式发现 — 维果茨基最近发展区

### 心理学原理
维果茨基 ZPD 理论:学习者在当前能力边缘获得支持时进步最大 — 不能太超前(焦虑),也不能太滞后(无聊)。随着学习者成长,"脚手架"逐渐撤除。

### 问题(邓宁-克鲁格效应)
新手不知道自己不知道。他们用搜索框做纯文本搜索,从未发现:
- `key:value` 语法(第 3 次使用 — 准备好学习了)
- 别名如 `st:pass`(第 8 次使用 — 掌握了语法,准备好用快捷方式)
- `/` 键盘快捷键(第 15 次使用 — power user 初现)
- 预设功能(第 25 次使用 — 专家就绪)

一开始就教全部 = 认知超载(违反 Sweller)。什么都不教 = 能力鸿沟持续存在(Dunning-Kruger)。解决方案: **在认知就绪的时刻教。**

### 洞察
在 localStorage 中追踪使用次数。到达预设阈值时,弹出一条一次性、可关闭的提示,在学习进阶中引入下一个特性:

```
第  3 次使用 → "💡 输入 字段:值 来筛选 — 例如 Status:Passing"
第  8 次使用 → "💡 别名如 st:pass 可以省击键"
第 15 次使用 → "💡 按 / 快速聚焦搜索框"
第 25 次使用 → "💡 点 ⭐ 保存常用搜索"
```

每条提示只出现一次,之后标记为已读。进阶过程映射维果茨基的脚手架:
1. **新手** (1-2 次): 自由文本 — 零学习成本
2. **初学者** (3-7 次): 发现结构化语法 — ZPD 脚手架
3. **中级** (8-14 次): 发现别名 — 效率提升
4. **高级** (15-24 次): 发现键盘快捷键 — 心流优化
5. **专家** (25+ 次): 发现预设 — 精通

### 实现
- `progressive.ts`: 使用计数器 + 提示调度器(localStorage 持久化)
- `hook.ts`: 每次搜索提交时 `incrementUsage()`, `getPendingHint()` 检查阈值
- `FilterChipBar.tsx`: 搜索框下方琥珀色提示条,可手动关闭
- 可定制: `hints` prop 允许应用覆盖默认进阶

### 为什么不做教程?
教程违反心流(Csikszentmihalyi) — 它打断主任务。渐进式发现是**情境化的**和**自定进度的**: 提示在实际使用中出现,而非在独立的学习场景中。用户在做中学,而非在读中学。

# 多协议前缀 — 频分复用 (FDMA)

### 通信原理
频分复用(FDMA)— 同一信道上,不同载波频率承载不同信号。

### 应用
当前只有一种"调制方案" `key:value`。但不同场景有不同的自然语法:

| 前缀 | 语法 | 场景 | 等价于 |
|------|------|------|--------|
| `#` | `#urgent` | 标签/分类 | `Tags:urgent` |
| `@` | `@zhang` | 人名提及 | `Assignee:zhang` |
| `>` | `>2024-01-01` | 日期比较 | `Date:>=2024-01-01` |

每种前缀是一种"调制方案",parser 根据**首字符**解调到不同的解析分支。`prefix` 字段让配置作者决定哪些字段支持前缀语法。
