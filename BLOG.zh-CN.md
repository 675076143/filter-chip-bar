# 一个搜索框的思考

> 你每天打开十个后台系统，面对十个不同的搜索区域。每个区域有下拉框、日期选择器、复选框、标签切换。你点击、选择、再点击、再选择。这不是搜索，这是填表。
>
> 有一天，你想：如果像 Google 一样，只给我一个框呢？

---

## 第一章：矛盾

后台系统的 UI 有一个 fundamental contradiction。

**结构化表单**精确但僵硬——下拉框告诉你"选这些"，你只能选这些。学习成本低，但天花板也低。想组合查询？想排除某个值？想用键盘完成一切？不可能。

**全文搜索**灵活但模糊——一个搜索框什么都搜，但你搜不到结构。`iPhone 15` 可能匹配到产品名，也可能匹配到描述里的某句话。你无法说"我要状态是通过的，订单量大于 100 的"。

这是黑格尔说的**正题与反题**。两个范式互相对立，各有缺陷。传统 UI 工程师的解法是**妥协**——既放搜索框，又放筛选控件。结果页面越来越长，认知负荷越来越重。

FilterChipBar 不妥协。它做的是**合题**。

---

## 第二章：合题

```
输入: 审核状态:通过 -部门:运营一部 订单量:>=100 kxccaqvx12
```

一行文字，五个条件。有结构化筛选（`审核状态:通过`），有排除（`-部门:运营一面`），有数值比较（`订单量:>=100`），有自由文本（`kxccaqvx12`）。全部在一个输入框里完成。

这不是"更好的搜索框"。这是对 UI 基本矛盾的**扬弃（Aufheben）**——保留正题的精确性，保留反题的灵活性，消除两者的僵硬和模糊。

用户不需要学 UI 布局。**想找什么，就输入什么。** 道家说"无为而无不为"——最好的 UI 是隐形的。当用户打字时，他不应该意识到自己在"操作一个组件"，他应该只是在**思考**，而系统在**回应**。

## 第三章：数学的温度

一个搜索框看起来简单。但简单的表面下，是六个学科的支撑。

### 图论：数据的血缘

部门 → 团队 → 项目。这不是三个独立下拉框，这是一张**有向依赖图**。选了工程部，团队列表应该只剩工程部的团队。FilterChipBar 的 `options` 可以是 `async (chips) => fetchTeams(chips['部门'])`——选项是图状态的函数，不是静态列表。

当用户在搜索框里输入 `部门:工程部 团队:前端`，第二个条件的选项自动被第一个条件过滤。这就是**电磁感应**——一个场的变化感应另一个场。法拉第发现了电磁感应，我们发现了筛选感应。

### 信息论：文字的压缩

`审核状态:通过` 需要 8 个字符。但信息熵只有 ~4 bit。能不能更短？

`st:pass`——6 个字符，同样的信息。这是**无损压缩**。`st` 是 `审核状态` 的别名，`pass` 是 `通过` 的别名。parser 是解压器，把短码还原成完整的筛选条件。

Unix 命令用缩写（`ls` 而非 `list`），DNS 用 CNAME，URL 用短链。同一个原理——**在信道带宽有限时（人类打字速度约 200 字符/分钟），提高编码效率。**

### 通信学：噪声中的信号

用户打字会犯错。`审核状态:通过` 打成了 `审核状态:通国`。传统组件只会标红——这是错误**检测**。FilterChipBar 做的是错误**纠正**：

> Did you mean **通过**?

这是**前向纠错（FEC）**——深空通信中用的技术。旅行者号向地球发信号，光速要 22 小时。不能要求"重传"。接收端必须自己纠正错误。Levenshtein 编辑距离 ≤ 2 时自动建议最接近的选项，用户一键纠正。

**92% 的打字错误是单字符错误。** 我们的"纠错容量"刚好覆盖。

## 第四章：人性的尺度

技术解决"能不能"的问题。心理学解决"好不好用"的问题。

### 邓宁-克鲁格的悬崖

新手不知道自己不知道。他用了 20 次搜索框，每次都是纯文本搜索。他不知道 `st:pass` 存在，不知道 `#标签` 存在，不知道预设功能可以一键恢复复杂查询。

教他？什么时候教？

第一天就教——认知超载（Sweller），吓跑了。
永远不教——能力鸿沟（Dunning-Kruger），他永远停留在新手。

维果茨基给出了答案：**最近发展区（ZPD）**。在用户"刚好准备好"的时候，给一个脚手架。

```
第 3 次使用 → 💡 输入 审核状态:通过 按字段筛选
第 8 次使用 → 💡 st:pass 是 审核状态:通过 的简写
第 15 次使用 → 💡 按 / 快速聚焦搜索框
第 25 次使用 → 💡 点 ⭐ 保存常用搜索
```

每条提示只出现一次。不打断，不强塞。用户在**使用中学习**，而不是在教程中学习。

### 心流的节拍

Csikszentmihalyi 说心流发生在挑战与能力匹配时。打字太快导致建议列表闪烁——心流断裂。打字太慢导致等待——心流也断裂。

FilterChipBar 的两个时间常数经过精心调校：

- **150ms**：输入框失焦后关闭下拉框的延迟——刚好够点击建议项
- **200ms**：级联筛选 re-fetch 的 debounce——刚好合并连续操作，用户感知不到延迟

这两个数字不是拍脑袋定的。人类的视觉反应时间约 200ms。低于这个阈值，大脑感知不到"等待"。高于 500ms，大脑开始焦虑。200ms 是**认知无感区**。

## 第五章：六个学科的支撑

一个搜索框。六个学科。

| 学科 | 贡献 |
|------|------|
| **数学** | 级联筛选（图论）、标签别名（信息论）、频率加权历史（贝叶斯概率） |
| **通信学** | FEC 模糊匹配、拥塞控制 debounce、多协议前缀（FDMA 复用） |
| **心理学** | 渐进式发现提示（ZPD 脚手架） |
| **物理学** | 最小作用量原理（设计哲学）、电磁感应（验证级联筛选） |
| **工程学** | SOLID 架构、安全系数、公差设计、反馈回路 |
| **哲学** | 黑格尔辩证法（合题）、奥卡姆剃刀（如无必要勿增实体）、道家无为 |

7 个代码特性，每个都有理论根基。3 个设计哲学，每个都经得起跨学科拷问。

这不是过度设计。每个特性都有明确的理论支撑,每个交互细节都有依据。

## 第六章：开源

这个组件叫 **filter-chip-bar**。

```bash
npm install filter-chip-bar
```

它是一个 headless hook（`useFilterChipBar`），也是一个 shadcn renderer，也是一个 antd6 adapter。你在 v1 用 antd4？写个 200 行的 renderer 就适配了。你在 v2 用 antd6？直接 import。你用 Tailwind + Radix？默认 renderer 就是 shadcn 风格。

它有 48 个单元测试，覆盖 parser、tokenizer、aliases、prefix、fuzzy matching。

它有完整的 Storybook 文档，中英双语，带 🌐 语言切换。

它有一个 Google 风格的官网——打开就能直接试。

它有 LICENSE、CHANGELOG、DESIGN.md（中英双语）、GitHub Actions CI/CD、npm 自动发版、Vercel Storybook 部署。

**GitHub**: https://github.com/675076143/filter-chip-bar
**npm**: https://www.npmjs.com/package/filter-chip-bar
**Storybook**: https://filter-chip-bar.vercel.app
